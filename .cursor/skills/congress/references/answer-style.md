# Стиль итогового ответа Congress

Обязательно для **`ANSWER.md`** (пишет **редактор**, фаза `editor`), **`ANSWER_PLAIN.md`** (ассистент) и раздела «синтез» в `synthesis/draft.md`.

## Главный принцип

Ответ — это **развёрнутое исследовательское заключение** с удобной вёрсткой: связная проза несёт аргументацию, **сводные таблицы и списки** — там, где читателю нужно быстро сравнить или пробежать глазами. Читатель должен понять *почему* комиссия пришла к выводу, *как* она спорила, *что* делать по шагам и *какие* риски принять.

**Объём:** ориентир **3 000–8 000 слов** для сложных задач; для простых — не меньше **1 500 слов**. Таблицы и списки — **не замена** текста целиком, но **обязательное дополнение** там, где уместна сводка.

## Жанр и тон

- Связная проза в духе **научно-популярного исследования** или **экспертного доклада**: введение → постановка → анализ → сравнение вариантов → выводы → рекомендации → ограничения.
- **Человеческий голос:** полные предложения, логические переходы; допустимы **жирный** и *курсив* для акцентов, названий вариантов, порогов и предупреждений.
- Уважительное «вы» к читателю. Без канцелярита, без рекламного пафоса.
- Каждый крупный раздел начинается с **абзаца-связки** (зачем этот раздел и что в нём узнает читатель).

## Язык

- **Основной язык — русский.** Английские слова и аббревиатуры **не вставлять** в середину русского предложения, если есть нормальный русский эквивалент.
- Допустимо **один раз** при первом упоминании: русский термин *(при необходимости — исходное обозначение в скобках)*, далее — только по-русски.
- Исключения (только если без них нельзя): устоявшиеся коды законов (152-ФЗ), официальные названия организаций, имена брендов и сервисов (Wildberries, Ozon, ФНС).
- **Запрещено** в основном тексте без пояснения: gate, track, approve, revise, ship, retainer, runway, compliance, lead, pitch, onboarding, stakeholder и т.п. — заменять русскими формулировками (см. глоссарий ассистента).

## Структура (обязательные разделы)

1. **Аннотация** (150–250 слов) — суть, вердикт, главная рекомендация одним связным текстом.
2. **Введение** — контекст задачи, что спрашивал пользователь, какие ограничения зафиксированы после intake.
3. **Метод** — как работала комиссия: кто участвовал, какие источники (исследования в интернете), как шёл диалог / маршрутизация в swarm.
4. **Анализ вариантов** — каждый серьёзный вариант **связным текстом**; удобный шаблон: абзац, начинающийся с **«Вариант N: …»** (жирным), затем плюсы/минусы в прозе или коротком подсписке; после блока вариантов — **сводная таблица сравнения**, если вариантов ≥2.
5. **Сравнение и аргументация** — где комиссары сошлись, где расходились; развёрнуто, с позициями ролей (**Критик**, *Юрист*…).
6. **Рекомендация** — что делать, в каком порядке, с обоснованием.
7. **Финансовая и операционная модель** — бюджет, сроки, точка безубыточности; цифры с пояснением *что они значат для читателя*.
8. **Риски** — для каждого риска желательно 1–2 предложения прозой; допустима **таблица** (вероятность / последствия / смягчение) после вводного абзаца.
9. **План действий** — по фазам: абзац «что делаете на этой неделе» + при необходимости маркированный чеклист.
10. **Ограничения исследования** — что не проверяли, что устареет, когда пересмотреть.
11. **Заключение** — итоговый абзац без новых фактов.

Для swarm-сессий добавить раздел **«Как формировалось решение»** — кто кого привлекал, какие запросы к исследователю меняли картину.

## Таблицы, списки и акценты

- **Сводные таблицы приветствуются** (сравнение вариантов, триггеры, риски, дорожная карта). Перед таблицей — 1–2 предложения; после — короткий вывод.
- Не начинать **весь раздел** с таблицы без вводного абзаца.
- Маркированные списки — в рисках, чеклистах, триггерах, перечне условий; не более **7 пунктов** подряд без связного абзаца до или после.
- Вложенные подпункты (плюсы/минусы варианта) — нормально, если вариант назван в начале абзаца.
- **Жирный** — вердикты, названия вариантов, пороги. *Курсив* — оговорки, dissent, «если контекст изменится».
- Не дублировать одну мысль в таблице и в тексте **дословно**: таблица сжимает, текст объясняет.

## ANSWER.md vs ANSWER_PLAIN.md

| | ANSWER.md | ANSWER_PLAIN.md |
| --- | --- | --- |
| Глубина | Полная, все нюансы комиссии | Та же глубина, проще лексика |
| Термины | Русские; редкие — с расшифровкой | Без нераскрытых аббревиатур; отсылки к `glossary/` |
| Объём | Не сокращать ради краткости | Не упрощать до «тезисов» |

Ассистент **не сокращает** ANSWER — он **переформулирует** для читателя без отраслевого жаргона, сохраняя полноту.

## Чего избегать

- Ответ **целиком** из таблиц и буллетов без связных абзацев между разделами.
- Одна строка вердикта без развёрнутого обоснования.
- Английские вставки: «делайте gate 14 дней», «условный approve», «productized offer».
- Сухая статистика без интерпретации: каждая ключевая цифра — с фразой «это значит, что…».

## Чеклист перед сохранением

- [ ] Есть аннотация и заключение связным текстом
- [ ] Каждый отклонённый вариант описан прозой (не только «отклонён»); при ≥2 вариантах есть сводная таблица или явные **Вариант N:** в тексте
- [ ] Нет английских слов в русских предложениях без необходимости
- [ ] Объём достаточный для «исследования», а не памятки
- [ ] Упомянуты источники из `research/findings/` где уместно
- [ ] Пройден `node congress/scripts/lint-answer.mjs ANSWER.md` (или `validate --gate full`)
