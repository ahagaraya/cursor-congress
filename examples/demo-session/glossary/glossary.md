# Глоссарий — SQLite или PostgreSQL

<!-- Термин → простое объяснение → зачем в этом отчёте -->

| Термин | Простое объяснение | Зачем в этом отчёте |
| --- | --- | --- |
| **SQLite** | База данных в одном файле на диске; приложение читает и пишет в неё напрямую, без отдельного сервера | Основной рекомендованный движок для минимально жизнеспособной версии в описанном контексте |
| **PostgreSQL** | Полноценный сервер базы данных: отдельный процесс, роли, сильная одновременная запись | Рассматривается как следующий шаг при росте, а не обязательная замена сейчас |
| **Минимально жизнеспособная версия (MVP)** | Первая рабочая версия продукта с минимальным набором функций для проверки идеи | Определяет масштаб задачи: один разработчик, мало пользователей, ограниченное время |
| **Сайд-проект** | Проект «для себя», не основная работа; здесь — с закрытым кругом тестеров | Объясняет, почему не нужна тяжёлая инфраструктура с первого дня |
| **Соло-разработчик** | Один человек пишет, разворачивает и поддерживает продукт | Главное ограничение: операционная нагрузка ложится на одного человека |
| **Виртуальный сервер (VDS/VPS)** | Арендованный удалённый компьютер в дата-центре, где крутится приложение | Весь сценарий построен вокруг одного такого сервера |
| **Персональные данные (ПДн)** | Любая информация, по которой можно узнать конкретного человека: имя, почта, телефон и т.п. | Меняют требования: база перестаёт быть «игрушечным файлом» |
| **152-ФЗ** | Федеральный закон о персональных данных в РФ | Задаёт рамку: локализация, правовое основание, политика, процедуры |
| **Обработчик персональных данных** | Тот, кто хранит, обрабатывает или передаёт персональные данные — вы или ваш провайдер | Важно при выборе управляемой базы: появляется ещё один обработчик |
| **Локализация данных** | Требование хранить персональные данные граждан РФ на серверах в России | Ограничивает выбор зарубежных управляемых сервисов и хранилищ копий |
| **Контур защиты персональных данных** | Минимальный набор мер: шифрование, резервные копии, проверка восстановления, политика, удаление | Условие, без которого SQLite не рекомендован |
| **Шифрование диска или тома** | Защита всего раздела диска паролем/ключом; без ключа файл базы не прочитать | Защищает файл SQLite при компрометации сервера или кражи диска |
| **Защищённое соединение (HTTPS)** | Шифрованная передача данных между браузером и сервером | Нужно, чтобы персональные данные не утекали «по дороге» |
| **Резервная копия (бэкап)** | Сохранённая копия базы на случай сбоя, ошибки или атаки | Без проверенных копий потеря данных почти неизбежна при инциденте |
| **Проверка восстановления** | Пробное развёртывание из копии на чистом сервере, чтобы убедиться, что она работает | Копия, которую никто не пробовал восстановить, — не страховка |
| **Режим журнала SQLite** | Способ работы SQLite, при котором чтение и запись могут идти одновременнее | Рекомендуется для веб-приложения с несколькими одновременными запросами |
| **Миграция базы данных** | Версионированное изменение структуры таблиц (схемы) | «Мелкие боли» SQLite; решаются переносимыми инструментами, а не сменой движка |
| **Слой доступа к данным** | Отдельный код между приложением и базой; запросы не размазаны по всему проекту | Делает будущий переход на PostgreSQL задачей на дни, а не на недели |
| **Транзакция** | Группа операций с базой, которая выполняется целиком или откатывается | Длинные транзакции в SQLite усиливают блокировки |
| **Блокировка базы** | Ситуация, когда одна операция записи мешает другим | Главный технический симптом, по которому пора переходить на PostgreSQL |
| **Одновременная (конкурентная) запись** | Несколько процессов или потоков пишут в базу в одно время | SQLite слабее PostgreSQL именно здесь; при росте — условие перехода |
| **Управляемая база данных** | База, которую обслуживает провайдер: обновления, копии, доступность | Вариант 3; допустим только с хранением в РФ и договором |
| **Разграничение доступа на уровне строк** | Правило «этот пользователь видит только свои строки в таблице» на уровне базы | Одно из условий перехода на PostgreSQL |
| **Роли в базе данных** | Разные учётные записи с разными правами: чтение, запись, администрирование | PostgreSQL даёт из коробки; SQLite — нет |
| **Политика конфиденциальности** | Документ для пользователя: что собираете, зачем, как храните, как удалить | Обязательная часть юридического контура |
| **Минимизация данных** | Собирать только те поля, без которых продукт не работает | Снижает риск и юридическую нагрузку |
| **Процедура удаления пользователя** | Понятный способ стереть данные человека по его запросу | Требование закона и здравого смысла при персональных данных |
| **Условие перехода (триггер)** | Конкретный признак, при котором пора менять решение | Защита от преждевременного перехода и от застревания на SQLite |
| **Операционная нагрузка** | Время и внимание на поддержку сервера, базы, копий, обновлений | Главная «цена» PostgreSQL для одного разработчика |
| **Секреты вне репозитория** | Пароли и ключи не лежат в коде на GitHub, а в переменных окружения или хранилище | Базовая гигиена; иначе утечка через репозиторий |
| **Субъект персональных данных** | Человек, чьи данные вы обрабатываете | Имеет право на удаление, доступ и обращения к вам |
| **Инцидент** | Сбой, утечка или подозрение на компрометацию данных | Нужна заранее описанная процедура реагирования |
